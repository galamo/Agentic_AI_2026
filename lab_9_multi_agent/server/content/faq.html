FAQ and general documentation

What is SSO?

Single Sign-On (SSO) lets users log in once and access multiple applications without entering credentials again. Our system uses a central identity provider.

How does login work?

Users authenticate via the identity provider. After success, they receive a session token. Permissions are loaded from the users and permissions tables and cached for the session.

Project guidelines

Use read-only database access for reporting. Only SELECT queries are allowed. For documentation questions, refer to this FAQ and the schema documentation.

What are permissions?

Permissions are named access rights (e.g. "admin", "view_reports"). They are linked to users via the users_permissions table. The audit_login table records login events.
